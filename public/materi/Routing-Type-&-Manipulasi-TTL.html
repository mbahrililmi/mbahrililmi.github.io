Pengantar

Apa itu Routing Type?

Ketika sebuah router menerima paket data, ia harus memutuskan apa yang akan dilakukan terhadap paket tersebut berdasarkan tabel routing. Normalnya, router akan meneruskan paket ke tujuan. Namun, MikroTik memberikan pilihan lebih lanjut — kita bisa mengatur tipe dari setiap entri route yang menentukan bagaimana router bereaksi terhadap paket tersebut.

Definisi Inti Routing Type adalah parameter pada entri route di MikroTik yang menentukan perilaku router terhadap paket yang cocok dengan destination address route tersebut. Ada empat tipe: Unicast, Blackhole, Prohibit, dan Unreachable.

Mengapa perlu tipe selain unicast? Dalam praktik nyata, administrator jaringan seringkali perlu memblokir akses ke subnet tertentu tanpa harus membuat firewall rule yang rumit. Routing type menyediakan cara cepat dan elegan untuk melakukan ini, sekaligus memberikan kontrol atas pesan error apa yang diterima pengirim.

Poin Penting Perbedaan utama antar tipe terletak pada dua hal: (1) apakah paket dibuang atau diteruskan, dan (2) apakah pengirim mendapat notifikasi ICMP atau tidak.

Lab 12

Routing Type Unicast

12

Unicast — Tipe Default

Paket diteruskan ke tujuan secara normal

Penjelasan Konsep

Unicast adalah tipe route yang paling umum dan merupakan default ketika kita menambahkan route baru. Ketika sebuah route bertipe unicast, artinya router akan meneruskan paket yang cocok menuju gateway yang ditentukan.

Nama "unicast" merujuk pada komunikasi satu-ke-satu: satu pengirim, satu penerima. Ini berbeda dengan broadcast (satu ke semua) atau multicast (satu ke banyak tertentu).

Cara Kerja PC1 mengirim paket ke 192.168.2.2 → R1 melihat tabel routing → menemukan route 192.168.2.0/24 via 12.12.12.2 → meneruskan paket ke R2 → R2 meneruskan ke PC2 → PC2 membalas → komunikasi berhasil.

Konfigurasi

Tambahkan route tanpa parameter type — otomatis menjadi unicast: 

// konfigurasi di R1/ip route add dst-address=192.168.2.0/24 gateway=12.12.12.2

Verifikasi Tabel Routing

// output: /ip route print# DST-ADDRESS        PREF-SRC   GATEWAY    DISTANCE
0 ADC 12.12.12.0/24  12.12.12.1 ether1     0
1 ADC 192.168.1.0/24 192.168.1.1 ether2    0
2 A S  192.168.2.0/24             12.12.12.2 1

Label A S berarti: Active + Static. Tidak ada huruf tambahan = unicast biasa.

Hasil Pengujian

// ping dari PC184 bytes from 192.168.2.2 icmp_seq=1 ttl=62 time=2.692 ms
84 bytes from 192.168.2.2 icmp_seq=2 ttl=62 time=2.288 ms
84 bytes from 192.168.2.2 icmp_seq=3 ttl=62 time=2.067 ms

Ping berhasil. Perhatikan ttl=62 — paket melewati 2 router (R1 dan R2), sehingga TTL berkurang 2 dari nilai awal 64.

Lab 13

Routing Type Blackhole

13

Blackhole — Silent Drop

Paket dibuang secara diam-diam, tanpa notifikasi apapun

Penjelasan Konsep

Bayangkan sebuah lubang hitam di luar angkasa — segala sesuatu yang masuk tidak akan pernah kembali dan tidak memberikan sinyal apapun. Itulah analogi dari tipe route Blackhole.

Ketika route bertipe blackhole, router akan membuang (drop) paket secara diam-diam. Pengirim tidak mendapat balasan apapun — bukan error, bukan timeout yang menjelaskan alasan, hanya keheningan total. Pengirim harus menunggu sampai timeout-nya sendiri habis.

Kapan Digunakan? Blackhole berguna ketika kita ingin memblokir traffic tanpa memberi tahu penyerang bahwa blokir sedang terjadi. Ini adalah teknik yang umum dalam keamanan jaringan untuk mencegah port scanning atau probing yang lebih lanjut.

Konfigurasi

Ubah tipe dari route yang sudah ada (entri nomor 2) menjadi blackhole: 

// mengubah tipe route di R1/ip route set 2 type=blackhole

Verifikasi

// output: /ip route print detail2 A SB dst-address=192.168.2.0/24 type=blackhole distance=1

Label sekarang A SB: Active + Static + Blackhole.

Hasil Pengujian

// ping dari PC1 — tidak ada respons192.168.2.2 icmp_seq=1 timeout
192.168.2.2 icmp_seq=2 timeout
192.168.2.2 icmp_seq=3 timeout
192.168.2.2 icmp_seq=4 timeout

PC1 hanya mendapat timeout — tidak ada penjelasan mengapa. Inilah ciri khas blackhole: silent drop.

Lab 14

Routing Type Prohibit

14

Prohibit — Administratively Prohibited

Paket dibuang + pengirim diberi tahu alasannya

Penjelasan Konsep

Berbeda dengan blackhole, tipe Prohibit tidak hanya membuang paket, tetapi juga mengirimkan pesan ICMP error kepada pengirim. Pesan ini menjelaskan bahwa akses diblokir secara administratif.

Pesan ICMP yang dikirim adalah: Type 3, Code 13 — yang artinya "Communication Administratively Prohibited". Ini seperti seorang satpam yang menolak masuk dan sekaligus memberitahu alasannya: "Dilarang masuk berdasarkan kebijakan manajemen."

ICMP Type 3, Code 13 Type 3 = Destination Unreachable. Code 13 = Communication Administratively Prohibited. Pesan ini dikirim langsung dari R1 (192.168.1.1) ke PC1, bukan dari tujuan akhir.

Konfigurasi

// mengubah tipe route menjadi prohibit/ip route set 2 type=prohibit

Verifikasi

// output: /ip route print detail2 A SP dst-address=192.168.2.0/24 type=prohibit distance=1

Label A SP: Active + Static + Prohibit.

Hasil Pengujian

// ping dari PC1 — mendapat pesan ICMP error*192.168.1.1 icmp_seq=1 ttl=64 time=0.756 ms (ICMP type:3, code:13,
  Communication administratively prohibited)
*192.168.1.1 icmp_seq=2 ttl=64 time=0.851 ms (ICMP type:3, code:13,
  Communication administratively prohibited)

Perhatikan bahwa sumber pesan ICMP adalah 192.168.1.1 (interface R1 yang terhubung ke PC1), bukan 192.168.2.2. Responnya juga sangat cepat karena tidak perlu menunggu timeout.

Lab 15

Routing Type Unreachable

15

Unreachable — Host Unreachable

Paket dibuang + pengirim diberi tahu host tidak terjangkau

Penjelasan Konsep

Tipe Unreachable mirip dengan Prohibit — sama-sama membuang paket DAN mengirim pesan ICMP. Namun, pesan yang dikirimkan berbeda. Unreachable mengirimkan: Type 3, Code 1 — artinya "Destination Host Unreachable".

Perbedaan semantisnya penting: Prohibit mengatakan "tidak boleh masuk karena kebijakan admin", sedangkan Unreachable mengatakan "host tersebut tidak bisa dijangkau (tidak ditemukan)".

Perbandingan Prohibit vs Unreachable Prohibit (Code 13) = blokir karena kebijakan. Ini lebih jujur — memberitahu bahwa ada blokir administratif. Unreachable (Code 1) = host tidak ditemukan. Ini bisa digunakan untuk menyembunyikan keberadaan host — seolah-olah host tersebut memang tidak ada.

Konfigurasi

// mengubah tipe route menjadi unreachable/ip route set 2 type=unreachable

Verifikasi

// output: /ip route print detail2 A SU dst-address=192.168.2.0/24 type=unreachable distance=1

Label A SU: Active + Static + Unreachable.

Hasil Pengujian

// ping dari PC1*192.168.1.1 icmp_seq=1 ttl=64 time=0.195 ms (ICMP type:3, code:1,
  Destination host unreachable)
*192.168.1.1 icmp_seq=2 ttl=64 time=0.851 ms (ICMP type:3, code:1,
  Destination host unreachable)

Sama seperti Prohibit, respons datang dengan cepat dari R1. Bedanya hanya pada kode ICMP yang dikirimkan.

Ringkasan

Perbandingan Keempat Tipe Route

Berikut adalah perbandingan lengkap keempat tipe route yang perlu dipahami:

Tipe Route Label Routing Drop Paket? Kirim ICMP Error? Pesan yang Dikirim
Unicast A S Tidak — Diteruskan Tidak perlu — (paket sampai ke tujuan)
Blackhole A SB Ya — Silent Drop Tidak dikirim — (pengirim tidak tahu)
Prohibit A SP Ya — Drop Ya — Dikirim ICMP Type:3, Code:13 (Admin Prohibited)
Unreachable A SU Ya — Drop Ya — Dikirim ICMP Type:3, Code:1 (Host Unreachable)
Cara Mudah Mengingat:
Unicast = teruskan. Blackhole = buang diam-diam (kata kuncinya: silent). Prohibit = buang + beritahu "dilarang masuk". Unreachable = buang + beritahu "tidak ditemukan". Semakin ke bawah, semakin "vokal" dalam memberikan respons kepada pengirim.
Lab 16

Routing Mark — Policy Based Routing

16

Routing Mark & Policy Based Routing (PBR)

Memilih jalur routing berdasarkan sumber paket

Penjelasan Konsep

Routing biasa (static atau dynamic) menentukan jalur berdasarkan tujuan (destination) paket. Dengan Policy Based Routing (PBR), kita bisa menentukan jalur berdasarkan kriteria lain — dalam lab ini, berdasarkan alamat sumber (source address).

Skenario: R1 memiliki dua jalur menuju R2 — via 12.12.12.0/24 dan via 21.21.21.0/24. Kita ingin PC1 (dari subnet 192.168.1.0) selalu menggunakan jalur 12.12.12.0, sedangkan PC2 (dari subnet 192.168.10.0) selalu menggunakan jalur 21.21.21.0.

Topologi Lab 16

PC1──192.168.1.0/24──R1──12.12.12.0/24──┐ | R2──192.168.2.0/24──PC3 PC2──192.168.10.0/24──R1──21.21.21.0/24─┘ Tujuan: PC1 (192.168.1.x) → jalur 12.12.12.0/24 PC2 (192.168.10.x) → jalur 21.21.21.0/24

Cara Kerja Routing Mark

Mekanismenya bekerja dalam tiga tahap:

  1. Mangle menandai (mark) paket berdasarkan source address. PC1 mendapat tanda "jalur-1", PC2 mendapat tanda "jalur-2".
  2. Tabel routing terpisah dibuat untuk setiap tanda. "jalur-1" memiliki default route via 12.12.12.2, "jalur-2" memiliki default route via 21.21.21.2.
  3. Rules IP Route mengarahkan paket bertanda tertentu ke tabel routing yang sesuai.

Konfigurasi IP Address (R1)

// konfigurasi alamat IP di R1/ip address add address=12.12.12.1/24   interface=ether1
/ip address add address=21.21.21.1/24   interface=ether2
/ip address add address=192.168.1.1/24  interface=ether3
/ip address add address=192.168.10.1/24 interface=ether4

Konfigurasi NAT

// NAT masquerade untuk kedua jalur keluar/ip firewall nat add chain=srcnat out-interface=ether1 action=masquerade
/ip firewall nat add chain=srcnat out-interface=ether2 action=masquerade

Verifikasi dengan Traceroute

// PC1 via 192.168.1.1 — melewati jalur 12.12.12.0/24/tool traceroute 192.168.2.2 src-address=192.168.1.1
1  12.12.12.2   ← via R2, jalur 12.12.12.0/24
2  192.168.2.2

// PC2 via 192.168.10.1 — melewati jalur 21.21.21.0/24/tool traceroute 192.168.2.2 src-address=192.168.10.1
1  21.21.21.2   ← via R2, jalur 21.21.21.0/24
2  192.168.2.2

Terlihat jelas bahwa paket dari subnet berbeda melewati jalur yang berbeda — inilah inti dari Policy Based Routing.

Manfaat PBR PBR sangat berguna untuk load balancing, quality of service (QoS), keamanan jaringan, atau ketika ISP yang digunakan berbeda untuk departemen yang berbeda dalam satu organisasi.
Lab 17

Manipulasi TTL dengan Mangle

17

Change TTL — Mengubah Time To Live

Manipulasi nilai TTL paket menggunakan Firewall Mangle

Apa itu TTL?

TTL (Time To Live) adalah nilai numerik dalam header paket IP yang berfungsi sebagai batas hidup paket di jaringan. Nilai ini mencegah paket beredar selamanya jika terjadi routing loop.

Mekanisme TTL Setiap kali paket melewati sebuah router, nilai TTL dikurangi 1. Jika TTL mencapai 0, router akan membuang paket dan mengirim pesan ICMP "Time Exceeded" ke pengirim. Nilai TTL default adalah 64 (Linux/MikroTik) atau 128 (Windows).

Ilustrasi Penurunan TTL

[PC1] [Router1] [Router2] [PC2] TTL=64 ───────→ TTL dikurangi 1 ───→ TTL dikurangi 1 ───→ TTL=62 TTL=63 keluar TTL=62 keluar Hasil ping: ttl=62 (berkurang 2 karena melewati 2 router)

Mengapa Perlu Mengubah TTL?

Ada beberapa alasan administrator mengubah nilai TTL:

  1. Membatasi jangkauan paket agar hanya bisa melewati sejumlah hop tertentu
  2. Mencegah tethering (berbagi koneksi) dengan membatasi TTL sehingga perangkat di balik hotspot terdeteksi
  3. Menormalkan TTL untuk keamanan (sembunyikan topologi jaringan internal)

Kondisi Awal

// sebelum konfigurasi — TTL normalPC1> ping 192.168.2.2
84 bytes from 192.168.2.2 icmp_seq=1 ttl=62 time=2.089 ms

Konfigurasi Mangle — Ubah TTL menjadi 10

// konfigurasi mangle di R1/ip firewall mangle add \
    chain=postrouting \
    out-interface=ether2 \
    action=change-ttl \
    new-ttl=set:10
Penjelasan Parameter chain=postrouting — aturan diterapkan setelah routing decision dibuat, saat paket hendak keluar. out-interface=ether2 — hanya berlaku untuk paket yang keluar dari ether2. action=change-ttl — aksi mengubah nilai TTL. new-ttl=set:10 — set = tetapkan nilai baru (bukan tambah/kurang), nilai barunya adalah 10.

Hasil Setelah Konfigurasi

// setelah konfigurasi — TTL berubah menjadi 10PC1> ping 192.168.2.2
84 bytes from 192.168.2.2 icmp_seq=1 ttl=10 time=1.773 ms
84 bytes from 192.168.2.2 icmp_seq=2 ttl=10 time=1.632 ms
84 bytes from 192.168.2.2 icmp_seq=3 ttl=10 time=1.463 ms

Nilai TTL berhasil diubah dari 62 menjadi 10. Koneksi tetap berjalan karena 10 hop masih cukup untuk mencapai tujuan dalam topologi ini.

Catatan Penting: Jika nilai TTL ditetapkan terlalu kecil (misalnya 1 atau 2) dan topologi membutuhkan lebih banyak hop, paket akan dibuang di tengah jalan dan koneksi akan gagal. Selalu sesuaikan nilai TTL dengan kebutuhan jaringan.
Penutup

Rangkuman Materi

Pada praktikum ini, mahasiswa telah mempelajari empat mekanisme penting dalam administrasi jaringan berbasis MikroTik:

  1. Routing Type (Unicast, Blackhole, Prohibit, Unreachable) — Memahami bagaimana router dapat dikonfigurasi untuk meneruskan atau memblokir paket, serta bagaimana memberikan atau menyembunyikan informasi error kepada pengirim.
  2. Policy Based Routing dengan Routing Mark — Kemampuan mengarahkan paket ke jalur berbeda berdasarkan sumber paket, bukan hanya berdasarkan tujuan.
  3. Manipulasi TTL dengan Mangle — Teknik mengubah nilai TTL paket untuk berbagai keperluan manajemen jaringan dan keamanan.
Yang Harus Dikuasai Mahasiswa diharapkan dapat membedakan kapan menggunakan setiap routing type, memahami pesan ICMP yang dihasilkan, dan mampu menerapkan konfigurasi tersebut pada skenario jaringan nyata.
