2 Materi Teori
Redundansi Jaringan ▼

Redundansi jaringan adalah konsep menyediakan jalur cadangan agar konektivitas jaringan tetap terjaga meskipun terjadi kerusakan atau gangguan pada jalur utama.

Redundansi sangat penting diterapkan pada:

  • Data Center – karena downtime berdampak besar pada bisnis
  • ISP (Internet Service Provider) – untuk menjamin SLA kepada pelanggan
  • Jaringan Perusahaan – terutama yang mengandalkan konektivitas internet 24/7
ℹ️ Dua mekanisme utama redundansi yang akan dipelajari adalah Fail Over (jalur cadangan aktif saat jalur utama down) dan Load Balancing (semua jalur aktif bersamaan untuk mendistribusikan traffic).
Fail Over ▼

Fail Over adalah mekanisme perpindahan otomatis dari jalur utama ke jalur cadangan ketika jalur utama mengalami gangguan (down).

Cara kerja Fail Over di MikroTik:

  1. Router menggunakan route dengan nilai distance TERENDAH sebagai jalur utama
  2. Route backup diberi nilai distance LEBIH TINGGI (contoh: distance=2)
  3. Parameter check-gateway=ping memonitor kondisi gateway secara berkala
  4. Jika ping ke gateway utama gagal, router otomatis beralih ke route backup
💡 Distance default pada static route adalah 1. Route dengan distance=1 akan diprioritaskan dibanding distance=2. Semakin kecil nilainya, semakin diprioritaskan oleh router.
Parameter Check-Gateway ▼

Parameter check-gateway pada route menentukan metode yang digunakan router untuk memonitor ketersediaan gatewayGateway: router atau perangkat yang menjadi pintu keluar menuju jaringan lain atau internet.

Nilai ↕ Cara Kerja ↕
ping Router melakukan ping ke gateway secara berkala. Jika ping gagal beberapa kali, gateway dianggap down dan route dinonaktifkan.
arp Router menggunakan protokol ARP untuk memeriksa ketersediaan gateway di jaringan lokal (layer 2).
(kosong) Tidak ada pengecekan. Fail Over TIDAK akan bekerja meskipun gateway benar-benar down.
⚠️ PENTING: Parameter check-gateway WAJIB dikonfigurasi pada jalur utama agar Fail Over berfungsi! Jika ada 3 jalur, check-gateway harus ada di jalur 1 dan jalur 2. Jalur terakhir (backup paling akhir) tidak perlu dikonfigurasi check-gateway.
Scope dan Target-Scope ▼

Dua parameter ini mengontrol bagaimana MikroTik menemukan next-hopNext-hop: router berikutnya yang harus dilewati untuk mencapai tujuan akhir dalam tabel routing:

  • scope: kemampuan jangkauan router saat mencari next-hop (default: 30)
  • target-scope: jarak maksimum yang diperbolehkan untuk mencapai gateway tujuan
💡 Analogi: Bayangkan scope adalah kemampuan jangkauan tangan kita, sedangkan target-scope adalah jarak ke tujuan. Jika target-scope > scope, kita tidak bisa menggapai tujuan tersebut secara langsung.

Parameter ini digunakan ketika gateway yang dituju tidak terhubung langsung ke router, sehingga perlu melewati router perantara untuk sampai ke gateway tersebut.

Load Balancing ▼

Load Balancing adalah teknik mendistribusikan beban traffic ke beberapa jalur secara BERSAMAAN. Berbeda dengan Fail Over, semua jalur aktif sekaligus untuk memaksimalkan penggunaan bandwidth.

Pada MikroTik, Load Balancing dilakukan dengan mendefinisikan multiple gateway dalam satu route. Rasio distribusi ditentukan dari berapa kali sebuah gateway disebutkan dalam konfigurasi.

Contoh rasio 2:1 (ISP1=20Mb mendapat 2x lebih banyak traffic dari ISP2=10Mb):

/ip route add dst-address=192.168.4.0/24 gateway=12.12.12.2,12.12.12.2,13.13.13.3
  • ISP1 (12.12.12.2) disebut 2 kali → mendapat 2/3 traffic
  • ISP2 (13.13.13.3) disebut 1 kali → mendapat 1/3 traffic
Perbandingan Fail Over vs Load Balancing ▼
Aspek ↕ Fail Over ↕ Load Balancing ↕
Penggunaan jalur Hanya jalur utama aktif Semua jalur aktif bersamaan
Efisiensi Bandwidth Rendah (1 jalur idle) Tinggi (semua jalur terpakai)
Kompleksitas Konfigurasi Sederhana Lebih kompleks
Ketersediaan (Availability) Tinggi Sangat tinggi
Cocok untuk Link kritis, koneksi penting Optimasi bandwidth, multi-ISP
3 Lab 8 – Konfigurasi Fail Over

Tujuan: Jalur via R2 sebagai prioritas utama. Jika R2 down, router otomatis beralih ke jalur via R3.

Topologi Jaringan:

──────── R2 ──────── / 12.12.12.0/24 \ 24.24.24.0/24 / \ R1 R4 ─── 192.168.4.0/24 ─── PC1 \ / \ 13.13.13.0/24 / 34.34.34.0/24 ──────── R3 ────────
ℹ️ Terdapat 4 router: R1 (sumber traffic), R2 (jalur utama atas), R3 (jalur backup bawah), R4 (tujuan akhir, terhubung ke PC1 di subnet 192.168.4.0/24).
1

Konfigurasi IP Address di semua router

R1 – dua interface ke R2 dan R3:

/ip address add address=12.12.12.1/24 interface=ether1 /ip address add address=13.13.13.1/24 interface=ether2

R2 – interface ke R1 dan ke R4:

/ip address add address=12.12.12.2/24 interface=ether1 /ip address add address=24.24.24.2/24 interface=ether2

R3 – interface ke R1 dan ke R4:

/ip address add address=13.13.13.3/24 interface=ether1 /ip address add address=34.34.34.3/24 interface=ether2

R4 – interface ke R2, R3, dan ke PC1:

/ip address add address=24.24.24.4/24 interface=ether1 /ip address add address=34.34.34.4/24 interface=ether2 /ip address add address=192.168.4.1/24 interface=ether3
2

Konfigurasi Static Routing di R1 (kunci Fail Over)

/ip route add dst-address=24.24.24.0/24 gateway=12.12.12.2 /ip route add dst-address=34.34.34.0/24 gateway=13.13.13.3 /ip route add dst-address=192.168.4.0/24 gateway=12.12.12.2 check-gateway=ping /ip route add dst-address=192.168.4.0/24 gateway=13.13.13.3 distance=2
📌 Penjelasan: check-gateway=ping memerintahkan R1 untuk secara berkala melakukan ping ke gateway 12.12.12.2 (R2). Jika ping gagal, route ini dinonaktifkan dan route backup (distance=2) via 13.13.13.3 otomatis aktif.
3

Konfigurasi routing di R2, R3, dan R4

R2:

/ip route add dst-address=13.13.13.0/24 gateway=12.12.12.1 /ip route add dst-address=34.34.34.0/24 gateway=24.24.24.4 /ip route add dst-address=192.168.4.0/24 gateway=24.24.24.4

R3:

/ip route add dst-address=12.12.12.0/24 gateway=13.13.13.1 /ip route add dst-address=24.24.24.0/24 gateway=34.34.34.4 /ip route add dst-address=192.168.4.0/24 gateway=34.34.34.4

R4:

/ip route add dst-address=12.12.12.0/24 gateway=24.24.24.2 /ip route add dst-address=13.13.13.0/24 gateway=34.34.34.3
4

Verifikasi tabel routing R1 kondisi normal

/ip route print

Kondisi normal: route ke 192.168.4.0/24 via 12.12.12.2 berstatus A S (Active Static) sebagai jalur utama. Route via 13.13.13.3 dengan distance=2 dalam keadaan standby (tidak aktif selama jalur utama hidup).

5

Pengujian traceroute kondisi normal (dari R1)

/tool traceroute 192.168.4.2

Hasil yang diharapkan: paket melewati R2, jalur: 12.12.12.2 → 24.24.24.4 → 192.168.4.2

6

Simulasi R2 down – jalankan perintah ini di router R2

/interface disable ether1

Amati tabel routing R1. Setelah beberapa detik (sesuai interval check-gateway), route utama via R2 akan dinonaktifkan secara otomatis dan route backup via R3 menjadi aktif.

7

Traceroute setelah failover terjadi

/tool traceroute 192.168.4.2

Hasil: paket sekarang melewati R3, jalur: 13.13.13.3 → 34.34.34.4 → 192.168.4.2. Fail Over berhasil!

4 Lab 9 – Fail Over WITHOUT Check-Gateway

Lab ini membuktikan secara langsung bahwa Fail Over TIDAK bekerja tanpa parameter check-gateway pada route utama.

1

Nonaktifkan route dengan check-gateway dan buat route baru tanpanya

/ip route disable [route yang menggunakan check-gateway] /ip route add dst-address=192.168.4.0/24 gateway=12.12.12.2

Kali ini route utama dikonfigurasi tanpa parameter check-gateway.

2

Simulasikan R2 down kembali (di R2) dan uji traceroute dari R1

/tool traceroute 192.168.4.2

Hasil: traceroute akan TIMEOUT! Router tidak mengetahui bahwa gateway sudah down sehingga tetap mengirim paket ke R2 yang sudah tidak aktif.

Kesimpulan Perbandingan:

Konfigurasi Route Utama Kondisi Saat Gateway Down Status Fail Over
Dengan check-gateway=ping Otomatis beralih ke jalur backup OK ✔
Tanpa check-gateway Tetap mengirim ke gateway mati → TIMEOUT GAGAL ✘
5 Lab 10 – Fail Over dengan Target-Scope
⚠️ Skenario Masalah: Link R1–R2 dalam kondisi baik, namun link R2–R4 mengalami gangguan (down). R1 tetap bisa ping ke R2 (gateway langsung), sehingga check-gateway biasa tidak mendeteksi masalah. Akibatnya Fail Over tidak aktif meskipun paket tidak dapat mencapai PC1.

Solusi: Gunakan target-scope dengan gateway yang tidak terhubung langsung ke R1 (yaitu 24.24.24.4 yang berada di balik R2).

1

Modifikasi route dengan mengubah gateway dan menambah target-scope

/ip route set [route] gateway=24.24.24.4 target-scope=30

Perubahan utama: gateway diubah dari 12.12.12.2 (terhubung langsung ke R1) menjadi 24.24.24.4 (tidak langsung, berada di sisi R4). Dengan begitu, check-gateway akan memonitor 24.24.24.4 yang akan gagal jika link R2–R4 down.

2

Simulasikan link R2–R4 down dan verifikasi failover

Nonaktifkan interface R4 yang mengarah ke R2, kemudian lakukan traceroute dari R1 ke PC1.

/tool traceroute 192.168.4.2

Hasil: karena gateway yang dimonitor adalah 24.24.24.4, ketika link R2–R4 down, ping ke 24.24.24.4 akan gagal dan Fail Over akan berfungsi dengan benar, beralih ke jalur via R3.

6 Lab 11 – Load Balancing

Skenario: ISP1 memiliki bandwidth 20 Mbps, ISP2 memiliki 10 Mbps. Traffic didistribusikan dengan rasio 2:1 sesuai kapasitas masing-masing ISP.

1

Nonaktifkan semua route Fail Over yang sebelumnya dikonfigurasi

/ip route disable [semua fail over routes ke 192.168.4.0/24]
2

Tambahkan route Load Balancing dengan rasio 2:1

/ip route add dst-address=192.168.4.0/24 gateway=12.12.12.2,12.12.12.2,13.13.13.3
ℹ️ Penjelasan Rasio: 12.12.12.2 ditulis 2 kali → mendapat 2/3 dari total traffic (proporsional dengan kapasitas 20 Mbps). 13.13.13.3 ditulis 1 kali → mendapat 1/3 traffic (sesuai kapasitas 10 Mbps). Total = 30 Mbps terdistribusi.
3

Verifikasi dengan traceroute berulang dari R1

/tool traceroute 192.168.4.2

Lakukan traceroute beberapa kali dan amati bahwa koneksi menggunakan kedua jalur (via R2 dan via R3) secara bergantian sesuai rasio yang dikonfigurasi.
