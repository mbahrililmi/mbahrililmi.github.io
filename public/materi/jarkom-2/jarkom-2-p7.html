2 Teori Load Balancing

Load Balancing adalah teknik mendistribusikan lalu lintas jaringan ke beberapa jalur/server secara merata untuk memaksimalkan penggunaan sumber daya, meningkatkan bandwidthKapasitas maksimum jalur data, diukur dalam Mbps/Gbps total, dan mencegah bottleneck. Dalam konteks dual-WAN, load balancing memungkinkan dua koneksi ISP digunakan secara bersamaan.

Metode Load Balancing MikroTik

🔀 ECMP (Equal Cost Multi Path) ▼

Cara termudah — tambahkan dua default route dengan distance yang sama. RouterOS akan membagi traffic secara otomatis berdasarkan destination address.

Kekurangan: Koneksi satu sesi bisa berpindah jalur (tidak persistent), bermasalah untuk HTTPS/VPN.

🔢 Nth (Round Robin) ▼

Mendistribusikan koneksi baru secara bergantian (round-robin) menggunakan counter. Setiap N paket/koneksi dikirim ke jalur berikutnya.

Contoh: Nth=2/2: koneksi ke-1 ke ISP1, ke-2 ke ISP2, ke-3 ke ISP1, dst.

🎯 PCC (Per Connection Classifier) ▼

PCC mengklasifikasikan koneksi berdasarkan hash kombinasi atribut (src-address, dst-address, src-port, dst-port). Satu koneksi selalu melewati jalur yang sama (sticky per connection) — mengatasi masalah ECMP.

  • Classifier: both-addresses, both-ports, dst-address-and-port, src-address-and-port
  • Per-Connection-Classifier=X/N: X = jalur ke berapa, N = total jalur
  • Cocok untuk dual-WAN dengan connection-sensitive protocols (HTTPS, VoIP, VPN)

3 Teori Failover

Failover adalah mekanisme perpindahan otomatis ke jalur cadangan ketika jalur utama mengalami gangguan. Tujuannya adalah memastikan ketersediaan layanan (high availability) meski ada kegagalan satu jalur.

👁️ Netwatch — Monitoring & Script Trigger ▼

Netwatch adalah fitur MikroTik yang melakukan ping berkala ke host tertentu. Jika host tidak merespons, Netwatch menjalankan skrip "down". Jika kembali merespons, menjalankan skrip "up".

  • Host: IP yang di-ping untuk mendeteksi status ISP (misal: 8.8.8.8)
  • Interval: Seberapa sering ping dikirim (mis: 10s)
  • Timeout: Berapa lama menunggu respons (mis: 5s)
  • Up Script: Dijalankan saat host kembali online
  • Down Script: Dijalankan saat host tidak merespons
🔀 Check-Gateway & Recursive Gateway ▼

check-gateway=ping: MikroTik secara aktif memonitor status gateway dengan mengirimkan paket ICMP (ping). Route dinonaktifkan otomatis jika gateway tidak merespons dalam waktu yang ditentukan. Ini adalah mekanisme dasar failoverPengalihan otomatis ke jalur cadangan saat jalur utama bermasalah di MikroTik.

💡 Check-gateway options:
  • ping — kirim ICMP ping ke gateway, tandai route inactive jika tidak ada reply
  • arp — gunakan ARP request (lebih ringan tapi hanya efektif untuk same-subnet gateway)
  • kosong — tidak ada pengecekan (route tidak otomatis dinonaktifkan)

🔁 Recursive Gateway — Penjelasan Mendalam

Recursive Gateway adalah teknik routing di mana gateway pada sebuah route ditentukan berdasarkan status keterjangkauan (reachabilityKemampuan untuk mencapai/menjangkau sebuah host atau jaringan) terhadap alamat target tertentu, bukan alamat gateway lokal secara langsung.

Mengapa perlu Recursive Gateway?

Permasalahan umum: ISP memiliki gateway lokal (misal 192.168.1.1) yang selalu merespons ping meskipun koneksi ke internet sebenarnya terputus (karena masalah di upstream ISP). Akibatnya, check-gateway=ping biasa tidak akan mendeteksi kegagalan ini.

Solusi dengan Recursive Gateway: Tambahkan route ke IP yang benar-benar ada di internet (misal 8.8.8.8) melalui gateway ISP, kemudian gunakan IP internet tersebut sebagai gateway efektif untuk default route.

Skenario TANPA Recursive Gateway:
  ISP-1 Gateway (192.168.1.1) masih merespons ping...
  ...tapi koneksi internet upstream ISP-1 putus!
  → check-gateway=ping TIDAK mendeteksi kegagalan ini
  → MikroTik tetap kirim trafik via ISP-1 → GAGAL!

Skenario DENGAN Recursive Gateway:
  Tambah route: 8.8.8.8 via 192.168.1.1 (scope=10)
  Default route: 0.0.0.0/0 via 8.8.8.8 check-gateway=ping (target-scope=11)
  
  → Jika 8.8.8.8 tidak terjangkau (internet ISP-1 putus):
     MikroTik mendeteksi 8.8.8.8 tidak reachable
     → Default route dinonaktifkan → trafik dialihkan ke ISP-2 ✅

Konfigurasi Recursive Gateway:

# Step 1: Tambah route ke IP Internet (8.8.8.8) via gateway ISP-1 # scope=10 = digunakan untuk "lookup" gateway, bukan untuk forward paket /ip route add dst-address=8.8.8.8/32 gateway=192.168.1.1 scope=10 # Step 2: Default route menggunakan 8.8.8.8 sebagai gateway recursive # target-scope=11 harus lebih besar dari scope route 8.8.8.8 (scope=10) /ip route add dst-address=0.0.0.0/0 gateway=8.8.8.8 \ target-scope=11 distance=1 comment="Primary via ISP-1 (Recursive)" # Step 3: Route backup via ISP-2 (langsung, non-recursive) /ip route add dst-address=0.0.0.0/0 gateway=192.168.2.1 \ distance=2 check-gateway=ping comment="Backup ISP-2"
✅ Kapan gunakan Recursive Gateway? Saat ISP Anda memiliki gateway lokal yang tetap aktif meskipun koneksi internetnya terputus. Recursive Gateway memastikan MikroTik benar-benar menguji konektivitas ke internet, bukan hanya ke gateway ISP lokal.

10 Tugas
  1. Implementasikan load balancing PCC dengan 3 ISP (ether1, ether2, ether3 sebagai WAN, ether4 sebagai LAN). Gunakan PCC 3 jalur (x/3 untuk x=0,1,2).
